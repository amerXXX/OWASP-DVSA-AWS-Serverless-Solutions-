# Lesson 10 · Unhandled Exceptions

## Vulnerability Summary
`DVSA-ORDER-MANAGER` accesses request body, Authorization headers, and JWT fields without checking whether they exist or are well-formed. Malformed requests crash the Lambda, returning HTTP 500 errors to the client and leaking internal exception details to CloudWatch logs.

## Root Cause
No defensive validation or centralized error handling:
- Function assumes `event.body`, `event.headers`, and `Authorization` always exist
- JWT is split by `.` without validating it has exactly 3 parts
- No `try/catch` blocks around `JSON.parse()`, JWT decode, or DynamoDB calls
- Error messages are generated by the framework (`"Internal server error"`) — not controlled by the app

## AWS Components
| Component | Role |
|-----------|------|
| Lambda (DVSA-ORDER-MANAGER) | Missing error handling |
| API Gateway | Returns raw 500 responses |
| CloudWatch Logs | Leaks stack traces |

## Tools Used
- `curl`
- `jq`
- CloudWatch Logs

## Reproduction Steps

### Example 1 — Missing Authorization Header
```bash
curl -X POST https://<API-ID>.execute-api.us-east-1.amazonaws.com/Stage/order \
  -H "Content-Type: application/json" \
  -d '{"action":"get"}'
```
**Result:** `{"message": "Internal server error"}` — HTTP 500

The function crashes trying to access `headers.authorization` which is undefined.

### Example 2 — Invalid JWT Format
```bash
curl -X POST https://<API-ID>.execute-api.us-east-1.amazonaws.com/Stage/order \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ABC" \
  -d '{"action":"get"}'
```
**Result:** `{"message": "Internal server error"}` — HTTP 500

Splitting `"ABC"` by `.` produces only 1 part; accessing `parts[1]` returns `undefined`, crashing `base64.decode()`.

## Evidence
Both attacks produce:
- HTTP 500 to the client — no useful error context
- CloudWatch logs contain the actual exception stack trace — internal details exposed to the log system

## Fix Summary
Add comprehensive input validation and centralized error handling:

```javascript
exports.handler = async (event) => {
  try {
    // Validate body exists
    if (!event.body || typeof event.body !== "string") {
      return { statusCode: 400,
        body: JSON.stringify({ status: "err", msg: "missing request body" }) };
    }

    // Validate Authorization header
    const authHeader = event.headers && event.headers.authorization;
    if (!authHeader || !authHeader.startsWith("Bearer ")) {
      return { statusCode: 401,
        body: JSON.stringify({ status: "err", msg: "missing or invalid token" }) };
    }

    // Validate JWT structure
    const parts = authHeader.split(" ")[1].split(".");
    if (parts.length !== 3) {
      return { statusCode: 401,
        body: JSON.stringify({ status: "err", msg: "invalid token format" }) };
    }

    // ... rest of processing ...

  } catch (err) {
    console.error("Unhandled error:", err);
    return { statusCode: 500,
      body: JSON.stringify({ status: "err", msg: "Internal error" }) };
  }
};
```

## Verification After Fix
| Input | Before Fix | After Fix |
|-------|-----------|-----------|
| Missing Authorization | HTTP 500 (crash) | HTTP 401 `"missing or invalid token"` |
| `Authorization: Bearer ABC` | HTTP 500 (crash) | HTTP 401 `"invalid token format"` |
| Valid request | Works | Still works |

## Lessons Learned
**Fail safely** — always validate input early, handle errors gracefully, and never expose internal implementation details to the client. In serverless systems, where multiple services interact and auto-scaling amplifies errors, proper error handling is critical for both security and reliability.
